## 1. Objetivo

Criar uma aplicação web simples em **Next.js**, hospedável na **Vercel**, para avaliar a capacidade técnica inicial de alunos de TI.

A avaliação deve ser curta, objetiva e prática, com cerca de **30 perguntas**, para não cansar o aluno e ainda gerar dados suficientes para classificar habilidades.

A aplicação deve:

- Coletar dados obrigatórios do aluno.
- Aplicar uma avaliação técnica curta.
- Avaliar habilidades por nota de 0 a 10.
- Gerar relatório individual.
- Criar um mind map visual das habilidades.
- Salvar cada avaliação como um JSON individual.
- Ter área `/admin` protegida por login simples.
- Permitir análise, busca, filtros e ranking geral.
- Não usar banco de dados tradicional.

---

## 2. Pesquisa e base de avaliação

A avaliação deve seguir uma lógica de **avaliação por competência**.

Não deve medir apenas se o aluno decorou conceitos. Deve medir:

- Conhecimento básico.
- Interpretação de código.
- Raciocínio lógico.
- Capacidade de resolver problemas.
- Noção prática de ferramentas.
- Clareza de comunicação técnica.
- Autopercepção do aluno.

Boas práticas a seguir:

1. Usar perguntas variadas, não só múltipla escolha.
2. Incluir leitura de código simples.
3. Incluir cenários práticos.
4. Incluir perguntas de autoavaliação, mas com peso menor.
5. Ter rubrica clara de pontuação.
6. Avaliar competências observáveis.
7. Manter a avaliação curta, com aproximadamente 30 questões.

### Observação importante sobre Vercel

A Vercel não garante persistência confiável em arquivos locais no ambiente serverless.

Portanto:

### Desenvolvimento local

Salvar os arquivos em:

```txt
/data/results
```

### Produção na Vercel

Usar preferencialmente:

```txt
Vercel Blob
```

Criar uma camada abstrata de storage:

```ts
interface StorageProvider {
  saveResult(fileName: string, data: AssessmentResult): Promise<void>;
  listResults(): Promise<AssessmentResult[]>;
  getResult(id: string): Promise<AssessmentResult | null>;
}
```

Implementar:

```txt
LocalFileStorageProvider
VercelBlobStorageProvider
```

## 6. Estrutura da avaliação

A avaliação deve ter aproximadamente **30 perguntas**.

Tempo estimado:

```txt
15 a 25 minutos
```

Não criar uma prova longa.

Distribuição recomendada:

| Área | Quantidade |
|---|---:|
| Lógica de programação | 5 |
| HTML e CSS | 4 |
| JavaScript | 5 |
| Banco de dados conceitual | 3 |
| Git e versionamento | 3 |
| Resolução de problemas | 4 |
| Organização técnica | 3 |
| Comunicação técnica | 3 |
| **Total** | **30** |

---

## 7. Tipos de pergunta

Usar os seguintes tipos:

```ts
type QuestionType =
  | "single_choice"
  | "multiple_choice"
  | "code_reading"
  | "scenario"
  | "scale"
  | "short_text";
```

### Pesos recomendados

| Tipo | Peso |
|---|---:|
| single_choice | 1.0 |
| multiple_choice | 1.2 |
| code_reading | 1.5 |
| scenario | 1.5 |
| scale | 0.6 |
| short_text | 1.0 |

Observação:

Perguntas de autoavaliação devem ter peso menor, porque o aluno pode se superestimar ou subestimar.

## 10. Modelo de pergunta

  max?: number;

## 11. Cálculo da pontuação

Para cada resposta:

```txt
score ponderado = score da resposta * peso da pergunta
```

Para cada habilidade:

```txt
nota da habilidade = soma dos scores ponderados / soma dos pesos das perguntas daquela habilidade
```

Pontuação geral:

```txt
pontuação geral = média das notas das habilidades
```

Arredondar para uma casa decimal.

Classificação:

```txt
0 a 3.9   = Iniciante
4 a 5.9   = Básico
6 a 7.9   = Intermediário
8 a 10    = Avançado
```

## 12. Correção de perguntas abertas

Perguntas abertas curtas devem ser corrigidas automaticamente por regra simples.

Para a pergunta Q29, procurar palavras-chave como:

```txt
erro
console
network
rede
api
requisição
dados
log
debug
testar
inspecionar
```

Regra sugerida:

- 0 palavras-chave: score 3
- 1 palavra-chave: score 5
- 2 palavras-chave: score 7
- 3 ou mais palavras-chave: score 9
- Se a resposta for muito clara e tiver passo a passo: score 10

## 20. Critérios de aceite

A aplicação estará pronta quando:

- `/` permitir avaliação sem login.
- O formulário exigir nome, email, telefone, curso e faculdade.
- Faculdade tiver apenas FICR e UCB.
- O teste tiver cerca de 30 perguntas.
- A avaliação misturar perguntas objetivas, cenários, leitura de código, escala e texto curto.
- Cada habilidade receber nota de 0 a 10.
- A pontuação usar pesos por tipo de pergunta.
- O resultado gerar nível final.
- O resultado gerar relatório automático.
- O resultado gerar mind map.
- Cada avaliação gerar JSON individual.
- O arquivo JSON tiver nome normalizado com aluno e timeframe.
- `/admin/login` aceitar admin/admin.
- `/admin` estiver protegido.
- Admin listar todos os testes.
- Admin permitir busca por nome.
- Admin permitir filtro por curso.
- Admin permitir filtro por faculdade.
- Admin mostrar ranking geral.
- Admin mostrar relatório individual.
- Admin mostrar gráficos.
- Layout ser moderno e responsivo.
- Projeto rodar localmente com `npm run dev`.
- Projeto estar preparado para deploy na Vercel.
